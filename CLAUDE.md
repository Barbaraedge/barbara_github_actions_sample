# CLAUDE.md

## Mission

This repository is a demonstrator for the Barbara platform/API lifecycle of a user application.

The goal is not to build a feature-rich app. The goal is to provide a small, predictable application artifact that can be used to demonstrate and validate this end-to-end flow:

1. generate a new application version
2. register that version in Barbara
3. deploy it to devices
4. verify that the deployment succeeded and the app is healthy
5. rollback to a previous version if validation fails

Treat this repository as an operational test artifact for versioning, deployment, validation, and rollback.

## Primary Goal

Preserve a minimal app that is:

- easy to version
- easy to package
- easy to deploy
- easy to verify on-device
- easy to correlate with a specific build and commit
- safe to rollback

## What This Repo Is

- a minimal Python HTTP app
- a deployment verification target
- a release/versioning demonstrator
- a CI/CD sample for Barbara-oriented workflows
- a place to validate traceability between version, artifact, deployment, and runtime health

## What This Repo Is Not

- a full product application
- a framework playground
- a place for unnecessary abstractions
- a place to add dependencies unless they clearly improve the demonstrator goal

## Agent Priorities

When making changes, optimize for these priorities in order:

1. preserve deployment and rollback demonstrability
2. preserve version traceability
3. preserve simple runtime verification
4. preserve CI stability
5. keep the codebase small and readable

## Critical Runtime Contracts

The app lives in `app/main.py` and serves on port `8080`.

Expected routes:

- `/`: returns `Everything works fine, version <full_version>`
- `/health`: returns `OK`
- `/favicon.ico`: returns `204 No Content`
- unknown paths: return `404`

These routes are not incidental. They exist to support deployment validation on devices.

## Versioning Contract

`VERSION` contains the human functional version, for example:

- `0.1.0`

The runtime version exposed by the app should be the functional version plus the short commit hash when available, for example:

- `0.1.0+17a0484`

Interpretation:

- `VERSION` identifies the release line or intended app version
- the short commit hash identifies the exact build source
- the combined value improves tracking across build, registration, deployment, verification, and rollback

Agent rule:

- do not hardcode commit hashes into `VERSION`
- the official deployment path is to inject the short hash during CI image build
- local git resolution is only a development fallback
- preserve the ability to run even when git metadata is unavailable

## Why The App Is Intentionally Simple

The application logic is intentionally minimal because the main subject under test is the Barbara deployment lifecycle, not business logic.

Complexity added to the app should only be accepted if it materially improves one of these:

- version traceability
- deployment realism needed by the demo
- health verification
- rollback confidence

## Important Files

- `app/main.py`: HTTP server and runtime version reporting
- `tests/test_app.py`: expected functional contracts
- `VERSION`: base functional version
- `Dockerfile`: container packaging
- `docker-compose.yaml`: local deployment and health verification
- `deployment-targets.json`: desired deployment target declaration by device and group name
- `.github/workflows/ci.yml`: test execution and image publishing on release tags
- `.github/workflows/release.yml`: release creation tied to CI and tags
- `scripts/barbara_api.py`: Barbara and Keycloak request helpers for release automation
- `scripts/resolve_deployment_targets.py`: deployment target validation and resolution before image build
- `README.md`: operational documentation for workflows, registry publishing, variables, and secrets

## Official Artifact Rule

The official built runtime artifact is the Docker image built and pushed by CI.

The final distributable artifact is a `.zip` generated by the release workflow. That archive must contain only one file:

- `docker-compose.yaml`

That `docker-compose.yaml` must point to the exact CI-built image using an immutable image tag derived from the short commit SHA.

Do not assume a manually rebuilt local image is equivalent to the official artifact used for deployment validation or rollback. The CI-built image carries the intended build metadata, including the embedded short commit hash, and the release zip must reference that exact image.

Registry publishing is driven by GitHub Actions configuration:

- repository variables: `REGISTRY_HOST`, `REGISTRY_NAMESPACE`, `IMAGE_NAME`
- repository secrets: `REGISTRY_USERNAME`, `REGISTRY_PASSWORD`

Environment-scoped Barbara and Keycloak settings are documented in `README.md` and must remain aligned with the workflows that consume them.

## Change Map

If you touch `app/main.py`:

- review `tests/test_app.py`
- preserve endpoint semantics unless the task explicitly changes the contract
- preserve the app's usefulness as a deployment probe

If you touch `VERSION`:

- preserve the distinction between base version and full runtime version
- consider impact on release tags, deployment tracking, and rollback reasoning

If you touch Docker files:

- preserve simple startup
- preserve health verification
- preserve CI injection of build metadata such as the short commit hash

If you touch GitHub workflows:

- preserve CI on `v*` tag pushes
- preserve deployment target validation before image build
- preserve generation of a resolved deployment target snapshot inside the validation job workspace for diagnostics
- preserve image build in CI with embedded short commit hash
- preserve release behavior tied to successful CI and `v*` tags unless the task explicitly changes release strategy
- preserve the final release zip as a single-file archive containing only `docker-compose.yaml`
- preserve the repo's role as an operational demonstrator

If you touch Keycloak-related files:

- preserve the integration between `scripts/barbara_api.py`, `scripts/resolve_deployment_targets.py`, `scripts/ensure_barbara_application.py`, and `scripts/upload_barbara_app_version.py`
- preserve useful diagnostics
- never leak secrets into source or logs

If you touch deployment target resolution:

- preserve validation against Barbara before Docker image build
- preserve exact matching of requested devices against `deviceName`
- preserve exact matching of requested groups against `name`
- preserve resolution of group devices through the group's `devices` field crossed with the company-wide device list
- do not reintroduce a dependency on Barbara endpoints that are known to be unstable for this flow unless they are explicitly fixed and revalidated

## Validation Checklist

Minimum validation before considering a change complete:

```bash
python -m unittest discover -s tests -v
```

If runtime behavior changes, also validate locally:

```bash
python app/main.py
```

Then verify:

```bash
curl http://localhost:8080/
curl http://localhost:8080/health
```

If packaging or startup changes, also validate:

```bash
docker compose up --build
```

## Behavioral Rules For Agents

- prefer the smallest change that preserves the demonstrator goal
- prefer standard library over new dependencies
- keep version reporting explicit and easy to inspect
- prefer CI-provided build metadata over runtime discovery when packaging artifacts
- keep responses deterministic and easy to assert in tests
- preserve compatibility with automated validation and manual device checks
- think about rollback impact before changing versioning or deployment-related behavior
- preserve the exact relationship between pushed image and release zip content

## Anti-Goals

Avoid these patterns unless explicitly requested:

- introducing a web framework
- adding configuration layers for hypothetical future needs
- making the app more complex without improving deployment validation
- changing endpoint messages casually
- coupling runtime behavior to git in a way that breaks packaged deployments
- weakening traceability between runtime app version and source commit
- relying on end users to manually inject tracking metadata for official deployment artifacts

## Good Contribution Definition

A good contribution in this repository usually does all of the following:

- strengthens or preserves deployment-demo usefulness
- improves traceability or operability
- keeps the app small and understandable
- updates tests when contracts change
- avoids unnecessary moving parts

## Working Assumption

Assume every code change should be evaluated through this question:

"Does this make the Barbara version-registration, deployment, validation, or rollback demo clearer, safer, or easier to operate?"

If the answer is no, the change is probably out of scope.
